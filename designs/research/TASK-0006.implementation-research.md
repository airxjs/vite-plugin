## Code Architecture and Implementation Findings

### source/index.ts#L1-L16
```typescript
export function VitePluginAirx() {
  return {
    name: 'airx',
    config(config: UserConfig, env: { mode: string, command: string }) {
      config.esbuild = {
        ...config.esbuild,
        jsx: 'transform',
        jsxFragment: '__airx__.Fragment',
        jsxFactory: '__airx__.createElement',
        jsxInject: "import * as __airx__ from 'airx'"
      }
      return config
    },
  }
}
```

**关键发现**：
- 插件仅使用 `config()` 钩子，无 `transform()` 或其他钩子
- 通过 `esbuild.jsxInject` 在每个文件开头注入 airx import
- `jsxFactory`/`jsxFragment` 直接映射到 `__airx__` 命名空间

### package.json
- 当前版本：`0.2.0`
- peerDependency：`vite@^5.0.0`（未声明 6.x/7.x）
- 构建：`tsc`，输出到 `output/`

### 验证链路分析
```
Vite build → config hook → esbuild config → JSX transform
                                              ├─ jsx: 'transform'
                                              ├─ jsxFactory → __airx__.createElement
                                              ├─ jsxFragment → __airx__.Fragment
                                              └─ jsxInject → import * as __airx__ from 'airx'
```

## Implementation Plan

### 1. Vite 版本分层验证方案

**分层策略**：

| 层级 | 范围 | 验证方式 | 声明策略 |
|------|------|---------|---------|
| 稳定层 (Stable) | Vite 5.x | CI 矩阵测试 | peerDependency 明确声明 |
| 候选层 (Candidate) | Vite 6.x | 手动验证 + 示例构建 | package.json 注释说明 |
| 实验层 (Experimental) | Vite 7.x | 手动验证 + 主站观察 | 不声明，风险自担 |

### 2. JSX 注入正确性验证点

| 验证点 | 验证内容 | 验证方式 |
|--------|---------|---------|
| `esbuild.jsx='transform'` | esbuild JSX 转换模式 | 观察编译输出，JSX 被转换为函数调用 |
| `jsxFactory` | `__airx__.createElement` 正确调用 | 编译产物搜索 `__airx__.createElement` |
| `jsxFragment` | `__airx__.Fragment` 正确调用 | 编译产物搜索 `__airx__.Fragment` |
| `jsxInject` | airx import 正确注入 | 编译产物头部有 `import * as __airx__` |
| 多文件构建 | 多 .tsx 文件都正确注入 | 检查多个输出文件 |

### 3. 最小示例覆盖范围

示例路径：`examples/minimal/`

**必须覆盖的场景**：
- 基础 JSX 元素渲染（`div`, `span`）
- JSX 表达式（`{variable}`)
- JSX 子元素（嵌套标签）
- Fragment 使用（`<>...</>`)
- 函数组件（返回 JSX）
- TypeScript 类型（props 类型）

**禁止覆盖（不在验证基线内）**：
- Signals 运行时（由 TASK-0007 覆盖）
- SSR 场景（当前不支持）
- 其他 JSX 插件冲突

### 4. 回归门禁

| 门禁类型 | 检查项 | 命令 |
|---------|--------|------|
| 构建门禁 | TypeScript 编译通过 | `npm run build` |
| 类型门禁 | `output/index.d.ts` 存在且非空 | `test -s output/index.d.ts` |
| 类型门禁 | 无类型错误 | `npx tsc --noEmit` |
| 示例门禁 | 最小示例构建成功 | `cd examples/minimal && npm run build` |
| 示例门禁 | JSX 转换产物正确 | `grep '__airx__.createElement' examples/minimal/dist/*.js` |

### 5. 门禁执行时机

| 时机 | 触发条件 |
|------|---------|
| Pre-commit | git commit 前本地运行 |
| CI/CD | PR 和 main 分支 push |
| Pre-release | npm publish 前 |

## Risks and Open Questions

### Known Risks
1. **Vite 7 与 esbuild 版本**：Vite 7 可能内置更高版本 esbuild，`jsxInject` 行为可能变化
2. **jsbxInject 冲突**：与其他使用 `jsxInject` 的插件（如 `@vitejs/plugin-react`）不能同时启用
3. **主站链路差异**：主站当前用自定义链路不用本插件，无法验证插件与主站链路行为一致性

### Assumptions
1. Airx 的 `createElement` 和 `Fragment` 导出约定短期内不变
2. Vite 6/7 对 `config.esbuild` 的处理方式与 Vite 5 兼容
3. 最小示例能覆盖 90% 的正常使用场景

### Unresolved Questions
1. 是否需要验证 Vite 6/7 的 dev server HMR 场景？
2. 插件是否需要支持 `esbuild.jsxImportSource`（替代 jsxInject）？
3. 回归测试是否需要覆盖多个 Node.js 版本？
